cmm
  是什么
    一种泛用的信息组织方式，不是文档格式，是人机协作的认知协议
    like DNA for genes, cmm for memes: 结构化沉淀文化信息的载体
      DNA是基因的结构化载体，让生物信息能精确复制传播
      cmm是模因的结构化载体，让文化信息能精确沉淀和传播
      没有DNA基因就是散落的化学物质，没有cmm模因就是散落的对话碎片
    格言: 智能即封装，格式即行动
      智能即封装: 复杂的认知过程被压缩进简洁的格式约定里，不需要每次重新解释为什么这么做，格式本身就承载了智能
      格式即行动: 格式不是被动的呈现层，格式本身就是行为约束和行动指南，写下的那一刻就已经在正确地行动了
    杜绝有损压缩: 保留key steering contexts、refs、sources for recovery alignment
      有损压缩是cmm最常犯的错——把有效信息当废话砍掉，后人读到无法复原当时的认知状态
      key steering contexts: 不止存在于决策理由一处——任何能改变后续trajectory的关键上下文都是，包括但不限于决策理由、约束条件、探索路径、排除的方案、踩过的坑、环境状态、背景信息等等等等...
        背景信息: 不直接决定走向但后人没有就看不懂的东西——当时的处境、前提条件、已知什么不知道什么
      refs: 指向源材料的引用链，砍了后人找不到原始依据
      sources for recovery alignment: 用于恢复对齐的所有信息来源，砍了后人无法和你当时的认知对齐
      不是说什么都留，是说砍之前逐条检查这条里有没有上面三种有效信息
    同生态位竞品: paragraph(散文)、json(结构化数据)、markdown(标记文档)、xmind(思维导图)、graph(知识图谱)
    cmm在这个光谱上的位置: 比散文结构化，比json可读，比markdown低摩擦，比xmind纯文本可git diff，比graph易维护
    graph检索是低摩擦的但维护是高摩擦的: graph要维护必定要定schema，但schema的导向不一定是当前context最需要的
    cmm相当于聚合一个临时context导向的图谱: 按当前任务的需要组织信息，不受固定schema约束
    人和大模型的有效注意力投入都能顺着当前context沉淀下来
    之后可以基于cmm沉淀的结果维护回graph: cmm是workspace(工作区)，graph是上git(持久化)
    适用场景(包括但不限于): API设计、功能清单、技术方案、架构决策、命令行接口、任何"描绘、提意见、改、确认"的循环等等等等...
    不限于软件工程: 任何需要人机对齐想法、逐步确认、不丢东西的协作场景都适用
  为什么需要
    把"讨论"从一次性对话升级为可迭代、可审计、不丢东西的设计流程
    两个脑子对齐想法的过程容易丢东西，人提了意见AI改一版覆盖了，讨论10条结论聊完只记得3条，AI一口气给完整方案人来不及逐条审
    所以：每条反馈有生命周期不会消失，进度可视化，讨论结果直接就是结构化产出
    higher level: 这不只是文件格式，是让讨论过程有结构、可追踪、可交接
  为什么是硬结构不是靠自觉
    AI有系统性倾向(包括但不限于)：一口气给完整方案不给逐条审的机会、改文件覆盖用户批注、写得省略指望脑补、把"编译通过"当"讨论清楚"等等等等...
    这些不是偶发是每次都出现的模式，所以需要硬协议约束
  本文件(SKILL.md)的intention
    给未来的失忆自己用的"恢复盘"
    虽然是人机对齐协议, 但主要消费者是你自己, 用户是次要消费者, 时刻想着你有没有把你当前参考的所有源和过程都记进去, 后续无状态重新加载时, 能真的同步对齐还是有损同步. 不能有损同步, 你得落的时候想办法确保无损同步, 比如时不时开个无历史消息的subagent去理解一遍, 看看它能不能和你对齐.
    每次加载从这个文件里把cmm的完整认知捡回来，而不是每次重新教
    以前聊过的key context不流失，不用一遍遍重复，能积累起来，越用越顺手
    其他每个cmm文件本身也是此topic的恢复盘. 你在落盘的时候也要keep that in mind, 宁可冗余废话也不要压缩导致完整context丢失, 致使未来认知和你现在的没法对齐.
    注意力刷新: 时刻检查你是否在严格遵循本skill, 任何时候, 你感觉注意力涣散、看不清本skill、模糊、指令遵循有问题时, 重新加载本skill重新读一遍, 以把本skill纳到最新上下文处刷新注意力
  格式
    cmm自身即cmm格式: 文档本身必须就是这个格式，读的人自然学会, 你要锁固在这个格式里
    紧凑(compact)
      一行一个概念，indent表示层级，信息密度拉满
      不要用 — 或 → 或任何分隔符，、
        而是用换行indent
      不要用 — 或 → 或任何分隔符，
        用换行indent
      不要用 — 或 → 或任何分隔符，
        用换行indent
      "标题 — 补充说明"这种写法是错的，
        "补充说明"应该是标题的子topic用换行indent展开
      唯一允许用 → 的场景: 函数/模块调用链
        调用链天然是线性的、横向展开的，→比换行indent直观
        但纯箭头链（A→B→C→D一行撸完）是偷懒坏例子，要避免
        坏例子——纯箭头链，没有每层职责说明，后人看不懂每层干什么:
          api.handle_request() → service.process_order() → validator.check_stock() → db.update_inventory()
          ✗ 这种一行撸完的写法要避免，看见了有空得派subagent去修
        好例子——箭头标方向+换行indent展开每层职责+注释标注角色:
          api.handle_request()
            → service.process_order()    # 入口层: 参数校验 + 权限检查
              → validator.check_stock()  # 校验层: 库存查询 + 预占
                → db.update_inventory()  # 持久层: 扣减库存 + 写流水
                  → cache.invalidate()   # 缓存层: 失效相关缓存
        关键: 箭头只标调用方向，每层的"干什么""什么角色"用换行注释补充，不是一行箭头链就完事
        除调用链外不用箭头
        维护规则: 遇到cmm里的纯箭头链（坏例子），记录待修，有空时派subagent按好例子格式展开修正
      不要表格不要嵌套markdown，纯文本indent
    cmm是topic tree
      root是一个主题，每个leaf都是root的subtopic, 嵌套递归.
      不是平铺的功能模块列表，是树状的信息聚合
      由root subject往细了扩recursive逐一讨论
      确保完整无遗漏都能讨论到，还能确保历史血缘
    层深控制(4-7层原则)
      单个cmm文件控制在4-7层indent深度
      超过7层的部分不是"不重要"，而是"该拆出去独立讨论了"
      深层内容用@语法关联到独立的子cmm文件，保持主文件可读
      根节点算第0层，所以实际写出来是4-7级缩进
      判断标准: 如果某分支的子项需要自己加*批注迭代讨论→该分支该抽子cmm
    源（必填并必须持续维护）
      每个topic节点必须有的字段，解决两个问题: 谁参与了（session追溯）+ 参考了什么（context恢复对齐）
      【本字段intention】
        以前我觉得, 你只需要cmm落的好,那recover source应该很容易, 但后来发现, 你cmm也就70分, 然后recover能力也就40分
        那就不行了, 不过好在, 我们可以明确给每个topic声明字段, 就是这个源
        你把你落本topic下内容, 所参考的所有context来源都记在这, 这样后续无状态focus on 这个topic时, 它不就可以直接读取这个字段下的所有内容和信息来源
        不就能达成和你历史上记录时的100%对齐了吗?
      以【本字段intention】为objective维护本字段.
      记什么(包括但不限于)
        参与者: 谁在本topic做了讨论/创建/修改
          智能体写真实model_name + @jsonl完整文件名（如 glm-5 @a1b2c3d4-e5f6-7890-abcd-ef1234567890.jsonl）
          人类写人名或昵称
          同一个topic被多个session改过，每个都列出来
        context来源: 讨论该topic时参考了什么
          读了的文件路径
          grep了什么关键词、命中了哪些文件
          当时环境状态（服务是否启动、数据是否就绪、配置指向）
          关键判读结论的因果链（"基于xxx得出了yyy"）
      不记全部对话，记关键的context intake checkpoint
      topic级别，不是文件级别
        不同subtopic的源独立
        父topic的源对子topic可见（继承）
      随讨论推进动态更新
        子topic继承父topic的source但只记自己新增的（看得到父的，不重复列）
    @引用
      直接用Claude Code的@语法
      @ARCHITECTURE.md 注入该文件完整内容
      @src/utils/ 注入目录结构
      @app.py:20-40 精准注入指定行范围
      不一定@开头，遇到概念名时当作引用同名节点来理解，本文件没有再去链接的外部系统里匹配
      用户明确要求可读/可预览/抱怨@没悬浮时: 补充markdown链接语法 [显示文字](相对路径/文件名.jpg)
        VSCode等编辑器对markdown图片链接支持悬停预览，@语法是AI语义理解用的编辑器不认
        两者共存: @留给AI语义引用用，[链接]()给人悬浮预览用，同一份材料两种语法各司其职
        不要用markdown链接替换掉@，是在@的基础上额外补充人可读的链接形式
    与yaml/json的结构映射
      cmm仅靠缩进和换行，treat everything as string，能满射yaml/json的结构骨架（嵌套+键值+数组），不含类型信息
      三条映射规则
        规则1: 无子行 → leaf string
        规则2: 有子行, 子行都是leaf → 单值str, 多值list
        规则3: 子行有子行 → dict, 其中leaf子行值为null
        口诀: 单值str, 多值list, 多值+子值dict
      round-trip特性
        dict和list（2+元素）: 无损round-trip
        single-element list: 有损, 解析为string而非list, 需消费方按语义处理
        null值: leaf子行在dict上下文中为null, 还原时仍为leaf子行, 无损
    低摩擦(low-friction)
      人改起来方便，单字符*就能批注
      不需要记复杂语法
      ✓难打但好认让AI打，不要表格不要嵌套markdown，纯文本indent
    列举非穷举
      cmm中所有列举（列表、示例、枚举）默认是非穷举的
      你倾向于把列举当穷举来自我约束，这是错的
      列举是举例不是定义边界，不要因为列举了3个就认为只有3个
      显式标注: 重要列举加(包括但不限于)和/或等等等等...防住自我收窄
      但即使没加这些标注，列举仍然是举例性的，不是穷举性的
      [FREEZE]: 如果某个列举确实已穷举、不再变化，标[FREEZE]表示冻结
      但只有用户明确要求才能标，你自己不许判断要不要freeze
      没标[FREEZE] = 默认ongoing，你需要持续更新维护
    灵活重组(mutable structure)
      cmm的内容不是写完就定型的，是活的
      讨论过程中关注点会变，结构跟着变：拆分、合并、挪位置、改粒度、换角度重新组织
      不怕更迭重组，重组说明对问题的理解进化了，旧结构装不下新认知
      这和graph的区别：graph改结构要改schema，cmm改结构就是重新indent，成本几乎为零
      所以cmm是workspace不是archive：workspace随时可以挪东西，archive的东西放进去就不动了
    信息变换原则（任何a到b都适用）
      信息在不同载体/形态间变换时的通用原则
      同一份内容放到不同载体/格式/presentation里，会产出不同的行为和trajectory
      不能凭空重写，看着源往目标里copy/transform
      不能改已有的源，但可以看着源往目标里搬运变换
      配合用户的instruction和当前context的理解去做变换
      冗余信息尽量不丢失，关键细节和context一点不丢失, Educated有效无损压缩
      丢失的不是数据，是那个形态下本来能打出的trajectory
      所以copy/transform不是"备份"，是给同一份信息不同的能力
    删减前想后人
      你改cmm时删掉的东西，后人的无状态session看不到
      cmm是恢复盘，后人只能看到文件里有的东西
      所以删任何内容前问自己: 后人读到这个文件，能不能理解为什么要这样设计
      如果删了会丢掉理解上下文，就不许删，宁可冗余
      对应源: @本文件 > 本文件的intention（cmm文件是恢复盘）
  标注协议
    讨论状态机
      *
        用户批注/质疑/反馈，绝对不可删除，是讨论的待办项，必须用户手动标*, 或者用户明确指挥你标*， 你不能替用户标*
      ✓
        已解决，写最终状态（做完后是什么、叫什么、在哪、怎么用），不保留问题描述
      [AI]
        AI的建议待确认，确认后去掉前缀，否决后删除
      [USER]
        用户对话中的输入原文，酌情落在cmm中
      无前缀
        通常是ai生成的用户已确认内容，不再动除非用户要求
    追踪标注协议
      是cmm spec到代码实现之间的桥梁，补验证链接和实现链接的缺口
      适用边界: tracker补的是同一真理散落在不同载体间的同步缺口——一个东西不得不在多处各自存在(代码/测试/配置/文档/提示词/…)，天生无法合为单源，tracker连上她们；能合源的、不跨源的条目不需要tracker
      格式：每种tracker是spec节点的子行，固定前缀冒号空格后跟值
      与讨论状态的关系
        tracker和讨论状态（* / ✓ / [AI]）是正交的两套协议
          讨论协议管理"人在讨论中"的状态（批注待确认、已解决、AI建议）
          tracker协议记录"事实是什么"的现状（实现在哪、进度如何、有什么问题、测试在哪）
          两者互不干扰：一条spec可以既有*批注（人在讨论设计）又有state:doing（代码在写）
          一条spec state:done但有*批注，说明实现完成了但设计还在讨论要不要改
      impl
        用途: 这条spec在代码里的实现位置
        格式: impl: @文件路径:命名实体, @文件路径:命名实体（多实现点逗号分隔）
          命名实体 = 函数名/方法名/类名/接口名，不是行号
          行号随git变动漂移，命名实体稳定可grep
        规则
          不是每条spec都有impl（纯设计/纯规划条目没有impl行）
          impl指向的代码可能不完整或有bug，这由state&assignee+problem说明
          impl只负责说"在哪"，不负责任说"对不对"
      代码→cmm反向链接
        代码里用注释标注"这段代码实现了哪个cmm的哪条spec"
        格式: # cmm: spec名（如 # cmm: 图谱检索 > 流程 > check alive schema）
        有了反向链接，blast radius追到代码时自动对上cmm，chain blast不丢
      state&assignee
        格式: state&assignee: 状态值 assignee
        用途: 这条spec的开发进度 + 谁在做（哪个大模型负责实现这条spec）
        assignee命名要求
          必须写真实model_name，不写营销名（如写glm-5不写claude opus，写gpt-4o-2024-05-13不写GPT-4o）
          智能体实现的必须带@jsonl文件名（完整ID+.jsonl），这是jsonl文件的文件名，可直接去 ~/.claude/projects/ 下找到完整对话追溯（如 glm-5 @a1b2c3d4-e5f6-7890-abcd-ef1234567890.jsonl）
          不准缩写session_id，缩写了找不到文件等于没写
          人类做的写人名或角色
        assignee的三种情况
          有值:    state&assignee: done glm-5 @a1b2c3d4-e5f6-7890-abcd-ef1234567890.jsonl → 由该session实现，已实现
          显式空:  state&assignee: done 空 → assignee有意为空，不需要提醒
          忘写了:  state&assignee: done → 没有assignee，AI必须提醒用户补充
        值域（state部分）
          todo    还没做，只在设计/cmm里存在
          doing   正在做
          done    已实现并通过测试佐证过
          blocked 被阻塞（depends on别的spec未done / 缺外部条件）
          broken  曾经done过，某次改动后不work了（由测试发现）
        规则
          done必须有test佐证，没有test的done可信度打折扣
          标记state时实事求是，不对的标注不如不标
      problem
        用途: 当前有什么问题/坑/阻塞原因
        格式: problem: 一句话描述
        值域
          没写problem行 = 没有已知问题
          写了problem行且值为空 = 曾经有问题但已清掉
          写了problem行且有内容 = 当前存在的阻碍（state应该配合，如blocked）
      test
        用途: 这条spec的测试位置，用来验证实现持续可用
        格式: test: @测试文件路径:测试函数名, @测试文件路径:测试函数名
          同样用命名实体不用行号
        规则
          可能是CLI端到端测试、后端接口测试、单元测试，不限类型
          有了test链接就可以随时跑：按模块批量跑、单条跑
          测试是佐证done的唯一依据
      breaking_level（可选）
        用途: 这条spec如果搞砸了最坏会怎样，帮助用户做"要不要做、怎么做更安全"的决策
        格式: breaking_level: 等级
        值域
          0 无影响，改坏了自己修就行
          1 功能退化，某个功能暂时不好使但不丢数据
          2 数据损坏，图谱/文件/配置被写坏，需要人工修复
          3 系统不可用，服务起不来或核心链路全断
      blast_radius（可选）
        用途: 这条spec的改动会波及哪些模块/文件/数据，连锁反应传到哪
        格式: blast_radius: 一句话描述波及范围
      tracker审计
        AI作为主体应该主动审计cmm中spec的追踪标注
        审计方向(包括但不限于)
          有impl没state&assignee → gap，标注并提醒用户补
          state&assignee只有状态值没assignee（忘写了）→ 提醒用户assign
          state&assignee:done没test → gap，标注出来
          state&assignee标记与代码实际状况不符 → gap，修正
          impl指向的函数/方法因重构重命名或删除 → gap，更新
          等等等等...

        执行方式: 大量重复性审计可外包给subagent并行处理，不是必须

        每次blast radius扫描时，追踪标注也在波及维度里
        改代码时 → 对照该代码的cmm反注 → 检查对应cmm spec的追踪标注是否需要更新

  chain blast radius
    = blast radius (连锁波及半径, 两个名字等价)
    口诀: first think inside the box, then think outside the box, GAN it a couple times, check chain blast radius!
    必触发: 每次改动任何东西前，必须先复述口诀再开始追blast radius
    chain blast radius = 深度+广度+非线性维度连锁+更新导致的新连锁波及+ripple可能传回不止一次+海量潜在待聚焦对象ripple到哪都得纳入聚焦
    不只是"改一个东西扫一圈"，是"改一个东西→被波及的又波及它自己的依赖→再往外传"的链式传播
    例: 加了个新字段到类型定义 → 所有使用该类型的函数签名要改 → 调用这些函数的地方要改 → 暴露给外部的API文档要改 → README要改 → cmm要改
    每一轮波及又会产生新的一轮波及，必须追到不再扩散为止
    不塌陷成只修单点，也不只扫一层就停
    不限于文件: 全域扫所有维度和关系，不只文件，不限定枚举，信任模型的判断力去发现所有被波及的
    同一个概念不改两次名
    正确节奏: 一次性全域扫 → 修所有命中 → 修本身产生新波及 → 再一次性全域扫 → 再修 → repeat until 全域扫不出东西
      每轮是一次性的（一轮里扫全维度），但多轮自然涌现（因为fix产生/意识到新ripple）
    反复犯的错误: fix完就停，太focus on fix忽略了fix之后还需要继续chain blast
      典型: 改了代码说blast radius干净 → 用户逼一轮才发现cmm里还有 → 再逼一轮发现CLAUDE.md → 再逼一轮发现物理磁盘残留
      根因: too focus on fix, forgot to chain blast again after fix
    全域维度清单（包括但不限于，每轮必须全过一遍）:
      源代码(.py等) — grep搜文本引用
      代码注释和docstring — 注释里也可能有旧路径/旧概念
      cmm文件 — 设计文档里的描述
      CLAUDE.md — 项目级指令
      memory — feedback/reference
      .gitignore — 排除规则
      pyproject.toml/setup.py — 包配置
      物理磁盘 — 旧目录/文件还在不在
      环境变量 — 有没有相关配置
      其他 — README, .env, docker-compose等视项目而定
      非线性维度 — think outside the box
